## Decisions

- Gates live in each product's `verify/`, not a shared `tools/website-gates/`: a
  scaffolded product carries its own gates and CI/verify.py discover them with no
  central edit (the `rule_testing` registration pattern). Divergence from the infra
  doc's sketch, recorded for the consistency win.
- Browser tier optional + SKIP-guarded (the lighthouse-not-a-dep precedent), so CI
  installs stay lean and the static tier alone gates every push.
- Proof subject is a clearly-labelled fiction (riso studio) over documenting a real
  tool, to keep full control of copy and fabricate nothing.
- Deferred (not silently cut): the inline-sprite icon strategy (inline SVGs are few
  + accessible; noted in `verify/README.md`).

## Strategic feedback

- **Worked:** the piece-by-piece branch -> PR -> CI-green -> squash cadence with a
  mutation-proven gate per increment kept every step shippable and auditable; the
  proof site being a visibly DIFFERENT studio's work (riso blue/magenta + grotesk +
  overprint, not the indigo/Fraunces example) is the real evidence the toolkit does
  not converge on a house look.
- **Suggestion:** a 7-piece build in one session hit CRITICAL context pressure near
  the end; split future builds across sessions (~2-3 pieces each) or run in a git
  worktree, and checkpoint at each piece boundary rather than only under pressure.
- **Health:** clean. main `aedef23`, 0 open PRs, both products build green, all
  gates mutation-proven, one friction annealed to a memory.
